# Operations on Canvas v1 — Formal Specification

Epic 35 | VIS-018 | Plan deliverable

## 1. Purpose

This document designs how **runtime/operations** appear on the visual canvas. Until now, the canvas represents the **design-time** model of a company (structure, capabilities, workflows, contracts, governance). This spec introduces a **runtime layer** that overlays operational state — workflow runs, incidents, and tasks — onto the existing design-time graph.

Operations are not a separate view. They are an **additive overlay** on the same canvas, consistent with the layer system established in VIS-002.

---

## 2. Design Principles

1. **Overlay, not reconstruction.** Runtime data appears on top of the design-time graph. It never replaces or restructures the underlying model.
2. **Attached, not floating.** Every runtime element is anchored to a design-time entity (workflow, stage, node, edge). There are no free-floating operational nodes.
3. **Toggle-able.** The runtime layer can be enabled/disabled like any other layer. When disabled, the canvas shows only the design-time model.
4. **Non-invasive to domain.** Runtime entities live in a separate bounded context (operations). They reference design-time entities by ID but do not extend or modify them.
5. **Polling v1, real-time later.** V1 uses standard REST polling. WebSocket/SSE for live updates is deferred.
6. **Temporal scope.** Runtime data has a time dimension. The canvas shows the "current" operational state by default, with optional time-range filters.

## 3. New Domain Entities

### 3.1 Run

A **Run** is a single execution instance of a Workflow.

```
Run {
  id:              string           // UUID
  projectId:       string
  workflowId:      string           // references Workflow.id
  status:          RunStatus        // 'pending' | 'running' | 'completed' | 'failed' | 'cancelled'
  triggerType:     RunTriggerType   // 'manual' | 'scheduled' | 'event'
  triggerDetails:  string | null    // free-text context about the trigger
  stageProgress:   StageProgress[]  // per-stage execution state
  startedAt:       string | null    // ISO timestamp
  completedAt:     string | null
  createdAt:       string
  updatedAt:       string
}

StageProgress {
  stageName:       string           // matches WorkflowStageDto.name
  stageOrder:      number           // matches WorkflowStageDto.order
  status:          StageStatus      // 'pending' | 'running' | 'completed' | 'failed' | 'skipped'
  assigneeId:      string | null    // references AgentAssignment.id (who is executing)
  startedAt:       string | null
  completedAt:     string | null
  output:          string | null    // free-text result or artifact reference
}
```

**Status transitions:**
```
pending → running → completed
                  → failed
                  → cancelled
pending → cancelled
```

### 3.2 Incident

An **Incident** is a problem or alert attached to any design-time entity.

```
Incident {
  id:              string           // UUID
  projectId:       string
  severity:        IncidentSeverity // 'critical' | 'warning' | 'info'
  status:          IncidentStatus   // 'open' | 'acknowledged' | 'resolved'
  title:           string
  description:     string
  targetType:      NodeType         // which type of entity this is attached to
  targetId:        string           // the entity ID
  relatedRunId:    string | null    // optional: caused by a run
  createdAt:       string
  resolvedAt:      string | null
  updatedAt:       string
}
```

**Status transitions:**
```
open → acknowledged → resolved
open → resolved
```

### 3.3 Task

A **Task** is a discrete unit of work assigned to an agent, typically generated by a Run or manually created.

```
Task {
  id:              string           // UUID
  projectId:       string
  title:           string
  description:     string
  status:          TaskStatus       // 'queued' | 'assigned' | 'in_progress' | 'completed' | 'failed'
  priority:        TaskPriority     // 'low' | 'medium' | 'high' | 'critical'
  assigneeType:    'agent-assignment' | 'role' | null
  assigneeId:      string | null    // references AgentAssignment.id or Role.id
  runId:           string | null    // optional: belongs to a run
  stageName:       string | null    // optional: belongs to a specific stage
  workflowId:      string | null    // optional: context workflow
  dueAt:           string | null
  createdAt:       string
  completedAt:     string | null
  updatedAt:       string
}
```

### 3.4 Entity relationship diagram

```
  Workflow (design-time)
     │
     │ 1:N
     ▼
    Run ──────── 1:N ──── StageProgress
     │                        │
     │ 0:N                    │ 0:1
     ▼                        ▼
   Task ◄─── 0:1 ──── AgentAssignment (design-time)
     │
   Incident ──── 0:1 ──── Any design-time entity
```

## 4. Visual Representation

### 4.1 New node types

| Node Type | Domain Source | Visual ID Pattern | Layer |
|-----------|-------------|------------------|-------|
| `run` | `RunDto` | `run:{id}` | Runtime |
| `incident` | `IncidentDto` | `incident:{id}` | Runtime |

Tasks are **not** rendered as independent canvas nodes. They appear as:
- Badges on agent-assignment/role nodes (showing queue depth)
- Rows in the inspector when selecting a run or agent

This avoids visual clutter — tasks are operational detail, not structural.

### 4.2 New edge types

| Edge Type | Source | Target | Style | Layer |
|-----------|--------|--------|-------|-------|
| `executes` | `run` | `workflow` | Solid, blue | Runtime |
| `at_stage` | `run` | `workflow-stage` | Dashed, blue | Runtime |
| `caused_by` | `incident` | `run` | Dashed, red | Runtime |
| `affects` | `incident` | Any design-time node | Dotted, red | Runtime |

### 4.4 Updated NodeStatus

The existing `NodeStatus` is extended with runtime-aware states:

```typescript
export type NodeStatus =
  | 'normal'       // no issues
  | 'warning'      // validation warning
  | 'error'        // validation error
  | 'dimmed'       // filtered out
  | 'active'       // currently executing (run in progress)
  | 'incident'     // has open incident
```

When the runtime layer is active, node status merges design-time validation status with runtime status. The worst status wins:
- `incident` > `error` > `active` > `warning` > `normal`

### 5.1 Layer definition

```typescript
{
  id: 'runtime',

  nodeTypes: ['run', 'incident'],

}
```

### 5.2 Visibility per zoom level

| Level | What the runtime layer adds |
|-------|-----------------------------|
| L1 | Incident badges on departments (count of open incidents). Run count badges on workflow-containing departments. |
| L2 | Active runs as small badges on workflow nodes. Incident nodes attached to any entity in scope. |
| L3 | Full run detail: run nodes inside workflow, `at_stage` edges highlighting the current stage, stage progress indicators. Incidents on stages. |
| L4 | Inspector shows full run history, task list, incident timeline. |

### 5.3 Default active layers updated

The runtime layer is **off by default** at all levels. Users explicitly enable it. This maintains the design-time-first principle.

```typescript
export const DEFAULT_LAYERS_PER_LEVEL: Record<ZoomLevel, LayerId[]> = {
  L1: ['organization'],
  L2: ['organization', 'capabilities'],
  L3: ['workflows'],
  L4: [],
}
// Runtime is never default — user opt-in
```

## 6. Canvas Rendering Strategies

### 6.1 L1 — Company Map with runtime

When runtime layer is active at L1:
- Each department node shows a **badge** with:
  - Active run count (e.g., "3 runs")
  - Open incident count with severity indicator
- No individual run/incident nodes at L1 (too dense)
- Department nodes with critical incidents get `status: 'incident'`

### 6.2 L2 — Department Map with runtime

When runtime layer is active at L2:
- Workflow nodes owned by the department show **run status badges** (e.g., "2 running, 1 failed")
- Incident nodes appear as satellite nodes near the affected entity
- Agent-assignment nodes show **task queue badges** (e.g., "5 tasks")

### 6.3 L3 — Workflow Map with runtime

This is the primary runtime view. When runtime layer is active at L3:
- **Run nodes** are rendered inside or below the workflow container
- The **current stage** of each active run is highlighted via `at_stage` edge
- Stage nodes show progress indicators:
  - Grey: pending
  - Blue pulse: running
  - Green check: completed
  - Red X: failed
  - Strikethrough: skipped
- Completed runs are collapsed by default (expandable)
- Incident nodes appear as red satellites on affected stages
- The latest N runs are shown (configurable, default 5)

### 6.4 Run detail (inspector, not canvas)

Selecting a run node opens the inspector with:
- **Overview tab**: status, trigger, timestamps, duration
- **Stages tab**: progress of each stage with status, assignee, timestamps
- **Tasks tab**: list of tasks generated by this run
- **Incidents tab**: related incidents
- **Output tab**: stage outputs and artifacts

## 7. Graph Projection Extension

### 7.1 Operations data source

Runtime data comes from a new **operations service** (or a new module within company-design-service). The graph projection service needs to optionally merge runtime data:

```typescript
class GraphProjectionService {
  async projectGraph(
    projectId: string,
    level: ZoomLevel = 'L1',
    entityId: string | null = null,
    requestedLayers: LayerId[] | null = null
  ): Promise<VisualGraphDto>
}
```

When `requestedLayers` includes `'runtime'`, the projection pipeline adds steps:

```
Existing steps 1-11 (unchanged)
12. if 'runtime' in activeLayers:
    a. runs = OperationsService.getActiveRuns(projectId, scope)
    b. incidents = OperationsService.getOpenIncidents(projectId, scope)
    c. runtimeNodes = mapRuntimeNodes(runs, incidents)
    d. runtimeEdges = mapRuntimeEdges(runs, incidents)
    e. nodes = [...nodes, ...runtimeNodes]
    f. edges = [...edges, ...runtimeEdges]
    g. nodes = applyRuntimeStatusOverlay(nodes, runs, incidents)
13. return graph
```

### 7.2 Runtime node mapping

```typescript
function mapRuntimeNodes(
  runs: RunDto[],
  incidents: IncidentDto[]
): VisualNodeDto[]
```

- Each active/recent run → one `run` node
- Each open incident → one `incident` node

### 7.3 Runtime edge mapping

```typescript
function mapRuntimeEdges(
  runs: RunDto[],
  incidents: IncidentDto[]
): VisualEdgeDto[]
```

- Each run → one `executes` edge to its workflow
- Each active run → one `at_stage` edge to the current stage
- Each incident with `relatedRunId` → one `caused_by` edge
- Each incident → one `affects` edge to its target entity

### 7.4 Runtime status overlay

When the runtime layer is active, design-time nodes get updated status:

```typescript
function applyRuntimeStatusOverlay(
  nodes: VisualNodeDto[],
  runs: RunDto[],
  incidents: IncidentDto[]
): VisualNodeDto[]
```

Rules:
- A workflow node with any active run → `status: 'active'` (unless has error/incident)
- A workflow-stage node that is the current stage of an active run → `status: 'active'`
- Any node targeted by an open incident → `status: 'incident'` (overrides all except existing `'error'`)
- A department with open incidents in its subtree → `status: 'incident'` (propagation)

### 7.5 Scope filtering for runtime nodes

Runtime nodes respect the same scope rules:
- **L1**: No runtime nodes rendered (only badges via status overlay)
- **L2**: Runs for workflows owned by the scoped department. Incidents targeting entities in scope.
- **L3**: Runs for the scoped workflow. Incidents targeting the workflow or its stages.

### 7.6 Badge data (aggregated counts)

For L1/L2 where individual runtime nodes are too dense, the projection adds **badge data** to existing nodes via a new optional field:

```typescript
export interface VisualNodeDto {
  // ... existing fields ...
  badges?: NodeBadge[]  // optional, only when runtime layer active
}

export interface NodeBadge {
  type: 'run-count' | 'incident-count' | 'task-count'
  count: number
  severity?: 'info' | 'warning' | 'critical'  // for incidents
  label: string                                 // e.g., "3 running", "1 critical"
}
```

## 8. REST Endpoints

### 8.1 Runs

```
GET    /projects/:projectId/runs                    → RunDto[]
GET    /projects/:projectId/runs/:runId             → RunDto
POST   /projects/:projectId/runs                    → RunDto (create/trigger)
PATCH  /projects/:projectId/runs/:runId             → RunDto (update status)
DELETE /projects/:projectId/runs/:runId             → void

GET    /projects/:projectId/workflows/:workflowId/runs → RunDto[] (runs for a workflow)
```

Query params for list: `status`, `workflowId`, `limit`, `offset`

### 8.2 Incidents

```
GET    /projects/:projectId/incidents               → IncidentDto[]
GET    /projects/:projectId/incidents/:incidentId   → IncidentDto
POST   /projects/:projectId/incidents               → IncidentDto
PATCH  /projects/:projectId/incidents/:incidentId   → IncidentDto
DELETE /projects/:projectId/incidents/:incidentId   → void
```

Query params for list: `status`, `severity`, `targetType`, `targetId`, `limit`, `offset`

### 8.3 Tasks

```
GET    /projects/:projectId/tasks                   → TaskDto[]
GET    /projects/:projectId/tasks/:taskId           → TaskDto
POST   /projects/:projectId/tasks                   → TaskDto
PATCH  /projects/:projectId/tasks/:taskId           → TaskDto
DELETE /projects/:projectId/tasks/:taskId           → void
```

Query params for list: `status`, `assigneeId`, `runId`, `priority`, `limit`, `offset`

### 8.4 Visual graph extension

No new visual-graph endpoint. The existing `GET /projects/:projectId/visual-graph` gains runtime support when `layers=runtime` is included:


## 11. Service Architecture

### 11.1 Where operations live

Operations entities (Run, Incident, Task) belong to a new **operations module** within `company-design-service`. They are a separate bounded context but share the same service for simplicity in v1.

```
services/company-design/src/
  operations/
    operations.module.ts
    domain/
      run.ts                    # Run aggregate
      incident.ts               # Incident aggregate
      task.ts                   # Task aggregate
      run-repository.ts         # interface
      incident-repository.ts    # interface
      task-repository.ts        # interface
    infrastructure/
      in-memory-run-repository.ts
      in-memory-incident-repository.ts
      in-memory-task-repository.ts
    application/
      run.service.ts
      incident.service.ts
      task.service.ts
      runs.controller.ts
      incidents.controller.ts
      tasks.controller.ts
    mapping/
      run-mapper.ts
      incident-mapper.ts
      task-mapper.ts
```

### 11.2 Integration with graph projection

`GraphProjectionModule` optionally imports `OperationsModule` to access runtime data when the runtime layer is requested.

The service only queries operations data when `'runtime'` is in the active layers, avoiding unnecessary work for design-time-only views.

### 11.3 Validation rules for operations

| Rule | Severity | Message |
|------|----------|---------|
| Run references non-existent workflow | error | "Run references unknown workflow {workflowId}" |
| Run stage progress references stages not in workflow | warning | "Stage '{name}' not found in workflow definition" |
| Incident targets non-existent entity | error | "Incident targets unknown {targetType} {targetId}" |
| Task assigned to non-existent agent/role | warning | "Task assignee {assigneeId} not found" |
| Run in 'running' status for >24h | warning | "Run has been running for over 24 hours" |

These validations extend `ValidationEngine` with a new rule set specific to operations.

| 2 | Should tasks be canvas nodes? | **No.** Tasks are inspector-level detail. Badges show counts. Too many tasks would clutter the canvas. |
| 3 | How many recent runs to show at L3? | **5 most recent** (configurable). Completed runs collapsed by default. |
| 4 | Should runtime layer affect snapshot/releases? | **No.** Releases capture the design-time model. Runtime is ephemeral and not versioned. |
| 5 | Should incidents propagate up the hierarchy? | **Yes, as badge counts.** A department shows aggregate incident count from all entities it contains. |
| 6 | Should Run aggregate validate workflow existence at creation? | **Yes.** Creating a run for a non-existent workflow returns 400. |
| 7 | Can the badges field on VisualNodeDto break existing consumers? | **No**, because it's optional (`badges?: NodeBadge[]`). Existing code that doesn't read badges is unaffected. |
